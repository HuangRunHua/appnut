# PRD-0003: 双模式部署（嵌入式 + 云端 C/S）

## 状态
草稿

## 背景

当前框架只有一种运行模式：嵌入式——Flux 引擎 + HTTP 服务 + 存储全部打包在 App 进程内，数据存在设备本地。这种模式适合单设备/离线场景，但无法支撑多设备同步、多用户协作等核心需求（WeChat 的多设备消息、Photos 的 iCloud 同步、任何需要"换手机数据还在"的场景）。

框架需要同时支持两种部署模式，让开发者根据 App 特性自行选择：

- **嵌入式模式**（现有）：适合离线工具、单设备 App
- **云端 C/S 模式**（新增）：服务端独立部署在云上，客户端通过网络访问

## 需求

### 云端 C/S 模式

1. **服务端独立部署**：Facet API + Admin API + 存储可以作为独立服务运行在云端（而非嵌入 App 进程），对外暴露 HTTP 端点
2. **客户端远程访问**：客户端 SDK（Swift/Dart/Rust）能指向远程服务端 URL，通过网络调用 Facet API，取代当前的 localhost 调用
3. **BFF 位置可选**：BFF（Flux 状态引擎 + handler）可以运行在客户端（当前方式），也可以运行在服务端（状态由服务端管理，客户端只做 UI 渲染）

### 模式切换

4. **配置驱动**：开发者通过配置（而非改代码）选择部署模式
5. **业务代码统一**：同一套 model + facet + BFF handler 代码，在两种模式下都能运行，不需要写两套
6. **Facet API 一致**：无论哪种模式，客户端调用 Facet API 的方式应一致（都是 HTTP 请求），只是目标地址不同（localhost vs 云端 URL）

### 基础数据同步（仅限云端模式）

7. **服务端作为数据权威**：在云端模式下，服务端存储是权威数据源，客户端每次请求都从服务端获取最新数据
8. **客户端缓存（可选）**：客户端可以缓存最近获取的数据以加速 UI 渲染，但不保证离线一致性（完整的离线优先+增量同步作为后续独立 PRD）

## 验收标准

- [ ] 同一个 TwitterFlux 示例能以嵌入式模式运行在 iOS App 内（现有行为不变）
- [ ] 同一个 TwitterFlux 示例能以云端模式部署：服务端跑在远程机器，iOS App 连接远程 URL
- [ ] 客户端 SDK 支持配置 `base_url`（嵌入式时为 localhost，云端时为远程地址）
- [ ] 两种模式下，Facet API 的请求/响应格式完全一致
- [ ] 文档说明两种模式的配置方式和适用场景

## 技术范围

- 涉及的 crate / 模块：`crates/bff/ffi`、`crates/dsl/client`、`example/rust/server`
- 需要修改的文件：FFI 初始化流程、客户端 SDK 的 URL 配置、服务端启动入口
- 需要新增的文件：可能需要独立的服务端启动入口（不经过 FFI）
- 客户端 SDK：Swift `FluxClient`、Dart `FluxClient` 需要支持远程 URL

## 约束

- 嵌入式模式的现有行为和性能不能退化
- 不在本 PRD 中实现完整的离线优先 + 增量同步（那是一个更大的课题，后续单独出 PRD）
- 云端模式下的存储后端仍使用现有的嵌入式存储（redb/sqlite），不在本 PRD 中引入 PostgreSQL 等外部数据库（后续可扩展）
